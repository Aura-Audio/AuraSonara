# 🏗️ SONARA MVP: Comprehensive Architecture & Code Plan

Transitioning from a high-level concept to a shippable MVP requires a rigorous engineering process. Below is the exact methodology to generate the plan, the updated hybrid architecture, a detailed file tree, and realistic development estimates.

---

## 1. The Process: How to Generate the Detailed Plan

To move from "idea" to "code," the engineering team must follow a strict 5-step documentation and design process. This prevents scope creep and ensures the audio engine remains performant.

1.  **Architecture Decision Records (ADRs):** Document *why* Rust + Faust was chosen over pure JS or C++. (e.g., "ADR-004: Use Faust for DSP nodes to reduce math errors and speed up MVP delivery").
2.  **API-First Design (OpenAPI/Protobuf):** Define the exact JSON schema for a `.sonara` project file and the WebSocket messages between the React UI and the Go Backend *before* writing UI code.
3.  **Audio Graph Modeling:** Map out the exact Web Audio API node graph. Define how audio flows from `MediaStreamAudioSourceNode` → `Rust Scheduler` → `Faust DSP Chain` → `Destination`.
4.  **Component-Driven UI Design:** Build the Timeline, Mixer, and Transport in isolated Storybook environments before integrating them into the main app shell.
5.  **Work Breakdown Structure (WBS):** Break every feature down into tasks that take no longer than 4 hours to complete, assigning LOC and time estimates to each.

## 2. Updated Hybrid Architecture (Rust + Faust + JS)

The biggest technical hurdle in a browser DAW is the **AudioWorklet**. It must run on a separate thread, cannot block, and must process audio in chunks of 128 samples. 

Here is the updated execution flow inside the browser:

```text

                          │
```

### Critical Path & Bottlenecks
1.  **The AudioWorklet Bridge (Weeks 4-6):** Passing audio buffers between the JS Main Thread, the JS Worklet Thread, and the Rust WASM module without copying memory (using `SharedArrayBuffer`) is the hardest technical challenge. If this fails, the MVP fails.
2.  **Canvas Performance (Weeks 8-12):** Rendering 50+ tracks with zooming, panning, and waveforms at 60FPS in React requires heavy optimization (offscreen canvases, `requestAnimationFrame` throttling).
3.  **Faust Parameter Smoothing:** Moving a EQ frequency knob in React must not cause "zipper noise" (audio clicks) in the Faust WASM node. Implementing linear interpolation in the Worklet is mandatory.

### Next Steps to Execute
1.  Initialize the Turborepo monorepo.
2.  Write the **ADR for the AudioWorklet Memory Model** (How JS and Rust will share audio buffers).
3.  Build a "Hello World" AudioWorklet that loads a Rust WASM file and outputs a sine wave. Once that works, the foundation is set.
