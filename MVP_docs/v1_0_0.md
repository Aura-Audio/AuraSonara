# 🚀 **SONARA MVP: Open-Source Development Plan**
**Target:** A fully functional, self-hosted, browser-based audio editing and mixing environment.
**Timeline:** 12 Months (52 Weeks) to Public v1.0 Release
**License:** AGPLv3 (Open Core)

---

## 1. Scope of the MVP (v1.0)
The MVP focuses strictly on **audio recording, arrangement, and mixing**. MIDI, VST hosting, and real-time collaboration are deferred to v1.1+ to ensure a stable, high-performance core.

**MVP Inclusions:**
*   Multi-track audio arrangement (linear timeline).
*   Browser-based recording (`getUserMedia`).
*   Non-destructive editing (split, trim, crossfade, move).
*   Basic mixer with routing (Tracks → Busses → Master).
*   5 core WASM effects (Gain, EQ3, Compressor, Delay, Reverb).
*   Local project saving (Browser File System API + Server fallback).
*   Stereo WAV/MP3 mixdown export.
*   One-click Docker deployment.

---

## 2. Technology Stack

| Layer | Technology | Rationale |
| :--- | :--- | :--- |
| **Frontend UI** | React, TypeScript, Zustand, TailwindCSS | Fast state management, component reusability. |
| **Timeline Renderer** | HTML5 Canvas API (via `react-konva` or custom) | High-performance rendering of hundreds of audio clips/waveforms without DOM lag. |
| **Audio Engine** | Rust (compiled to WASM) + Web Audio API | Rust provides memory-safe, near-native DSP performance; AudioWorklets prevent UI thread blocking. |
| **Backend API** | Go (Gin or Fiber) | Extremely low memory footprint, high concurrency for file I/O and WebSockets. |
| **Storage/DB** | SQLite + Local File System | Zero-config for self-hosters. SQLite handles project metadata; FS handles heavy `.wav` blobs. |
| **DevOps** | Docker, Docker Compose, GitHub Actions | Ensures identical environments from dev to production VPS. |

## 3. Development Stages (The 12-Month Roadmap)

### **Phase 0: Architecture & Foundation (Weeks 1–6)**
*Goal: Establish the monorepo, CI/CD, and basic client-server communication.*
*   **Deliverables:**
    *   Monorepo setup (Turborepo) with `apps/web`, `apps/server`, `packages/dsp-wasm`.
    *   Docker Compose configuration for local development (Frontend + Backend + SQLite).
    *   Basic Web Audio API wrapper (playing a static audio file in the browser).
    *   Rust-to-WASM build pipeline configured.
*   **Exit Gate:** A user can run `docker compose up`, open `localhost:3000`, and hear a test tone generated by the Rust WASM module.

### **Phase 1: The Audio Engine & Transport (Weeks 7–14)**
*Goal: Build the underlying playback and recording engine.*
*   **Deliverables:**
    *   **AudioWorklet Node:** Custom worklet for sample-accurate playback and processing.
    *   **Transport Controls:** Play, Stop, Pause, Record, Loop, and Timeline Scrubbing.
    *   **Recording Engine:** Capture microphone input via `getUserMedia`, buffer it, and write it to a temporary WAV file in memory.
    *   **Waveform Generation:** WebWorker that analyzes audio buffers and generates peak data for UI rendering.
*   **Exit Gate:** User can record audio from their mic, stop, and immediately play it back with sample-accurate sync.

### **Phase 2: Timeline & Arrangement UI (Weeks 15–24)**
*Goal: Create the visual workspace for arranging audio clips.*
*   **Deliverables:**
    *   **Canvas Timeline:** Render tracks vertically and time horizontally. Implement smooth zooming and panning.
    *   **Clip Management:** Drag-and-drop audio clips. Implement visual trimming (edges) and splitting (razor tool).
    *   **Snapping & Grid:** BPM-based grid lines with snap-to-grid functionality.
    *   **State Management:** Zustand store to track clip positions, track heights, and zoom levels.
*   **Exit Gate:** User can import 3 audio files, arrange them on 3 tracks, split them, and move them around smoothly at 60fps.

### **Phase 3: The Mixer & Routing Graph (Weeks 25–32)**
*Goal: Implement the digital mixing console.*
*   **Deliverables:**
    *   **Routing Matrix:** Backend logic to route audio from Track Nodes → Bus Nodes → Master Node.
    *   **Mixer UI:** Channel strips with vertical faders, pan knobs, mute/solo buttons, and phase invert.
    *   **Metering:** Real-time peak and RMS metering on every channel strip (calculated in WASM, sent to UI via `MessagePort`).
    *   **Automation Basics:** Simple volume automation lanes (drawn on Canvas).
*   **Exit Gate:** User can adjust faders, pan audio left/right, and see real-time metering react to the audio.

### **Phase 4: DSP Effects & Processing (Weeks 33–40)**
*Goal: Port essential studio effects to WASM.*
*   **Deliverables:**
    *   **Insert Slots:** UI and engine support for chaining up to 5 effects per track.
    *   **Core Effects (Rust/WASM):**
        1.  *Utility:* Gain, Phase, Polarity.
        2.  *EQ:* 3-Band Parametric EQ.
        3.  *Dynamics:* Basic Compressor (Threshold, Ratio, Attack, Release).
        4.  *Time:* Stereo Delay.
        5.  *Space:* Algorithmic Reverb (based on Freeverb or similar open-source algo).
*   **Exit Gate:** User can apply an EQ and Compressor to a track, tweak parameters via UI knobs, and hear the DSP processing in real-time with <15ms latency.

### **Phase 5: Project Management & Export (Weeks 41–48)**
*Goal: Ensure users can save their work and export the final product.*
*   **Deliverables:**
    *   **Project Schema:** JSON structure defining tracks, clips, mixer settings, and plugin states.
    *   **File System Access API:** Allow users to save/load the project folder directly to their local hard drive (Chrome/Edge).
    *   **Server Fallback:** Upload/download project archives (`.sonara` zip files) for browsers that don't support local FS access.
    *   **Bounce/Export:** Offline rendering engine that processes the entire timeline and exports a 16/24-bit WAV or 320kbps MP3.
*   **Exit Gate:** User can save a complex project, close the browser, reopen it, load the project, and export a final stereo mix.

### **Phase 6: Polish, Optimization & Beta (Weeks 49–52)**
*Goal: Prepare for public release.*
*   **Deliverables:**
    *   **Performance Profiling:** Memory leak hunting, WASM optimization, reducing main thread blocking.
    *   **UI/UX Polish:** Dark mode refinement, keyboard shortcuts (J-K-L, spacebar, ctrl+z), tooltips.
    *   **Documentation:** Setup guides, user manual, and API docs (using MkDocs/Docusaurus).
    *   **Closed Beta:** Invite 50-100 users from r/selfhosted and r/WeAreTheMusicMakers for stress testing.
*   **Exit Gate:** Zero critical bugs, memory usage stable over 2-hour sessions, positive beta feedback.

## 4. Open-Source & Community Strategy

To ensure the project gains traction and contributors, the open-source strategy must be intentional from Day 1.

1.  **Repository Hygiene:**
    *   Comprehensive `README.md` with GIF demos and 1-click deploy buttons (e.g., "Deploy to DigitalOcean").
    *   Strict `CONTRIBUTING.md` and `CODE_OF_CONDUCT.md`.
    *   Use of `good first issue` and `help wanted` labels.
2.  **Community Channels:**
    *   **Discord/Matrix:** For real-time dev chat and user support.
    *   **GitHub Discussions:** For feature requests and architecture debates (keep issues strictly for bugs/tasks).
3.  **Documentation-Driven Development:**
    *   Write the architecture docs *before* coding complex features. This allows community members to review the design and suggest improvements before code is written.
4.  **Plugin Architecture (Future-proofing):**
    *   Even if not fully exposed in MVP, design the WASM audio nodes so that third-party developers can eventually write their own Rust/C++ effects and compile them to WASM to drop into Sonara.

## 5. Resource Requirements (Core Team)

For a 12-month MVP, a lean core team of 3-4 people is ideal:

*   **1x Audio/DSP Engineer (Rust/C++):** Focuses purely on the WASM audio engine, effects, and Web Audio API integration.
*   **1x Frontend/UX Engineer (React/Canvas):** Focuses on the timeline, mixer UI, state management, and performance.
*   **1x Backend/DevOps Engineer (Go/Docker):** Focuses on file I/O, database, authentication (for future multi-user), and deployment scripts.
*   **1x Product/Community Manager:** Handles beta testing, documentation, roadmap prioritization, and community engagement.

## 6. Risk Management

| Risk | Impact | Mitigation Strategy |
| :--- | :--- | :--- |
| **Browser Audio Latency** | High (Unusable for recording) | Utilize `AudioWorklet` strictly. Implement a configurable buffer size slider (e.g., 128 to 1024 samples) so users can balance latency vs. CPU load based on their machine. |
| **Memory Leaks (WASM)** | High (Browser crashes) | Implement strict Rust memory management. Use tools like Chrome DevTools Memory Profiler weekly. Avoid passing large audio buffers between JS and WASM; use `SharedArrayBuffer` where possible. |
| **Safari/iOS Incompatibility** | Medium (Limited user base) | Web Audio API on iOS Safari is notoriously restrictive (requires user gesture to start). Implement a prominent "Click to Enable Audio" overlay. Defer full iOS support to v1.1. |
| **Scope Creep** | High (Delayed launch) | Ruthlessly defer MIDI, VSTs, and Collaboration. If a feature doesn't directly enable recording, editing, or mixing audio, it goes into the v1.1 backlog. |

## 7. Definition of Done (MVP Launch)

The MVP is considered "Done" and ready for public GitHub release when:
1.  [ ] A user can deploy Sonara on a $5/mo VPS using a single `docker compose up -d` command.
2.  [ ] The user can record 4 tracks of audio simultaneously (or via overdub) without audio drift.
3.  [ ] The user can apply at least 3 effects per track without the audio engine dropping out or glitching.
4.  [ ] The user can export a final stereo mix that sounds identical to the timeline playback.
5.  [ ] The project has passed a basic security audit (no exposed directories, secure websocket connections, sanitized file uploads).
